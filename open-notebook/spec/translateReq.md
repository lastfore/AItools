请严格按照以下要求，将我提供的英文技术文档翻译成中文。

【翻译要求】
1. **术语准确性**  
   - 技术术语必须使用行业通用、权威的中文译法（如 database → 数据库，API → 应用程序接口/API）。  
   - 对于尚无统一译名或容易引起歧义的术语，首次出现时保留英文原词并用括号标注中文说明，例如：Rebase（变基）。  
   - 保持全文术语一致，同一英文术语不应出现多种译法。

2. **格式与标记保留**  
   - 完整保留原文的 Markdown 标记、HTML 标签、代码块、命令行、表格结构、列表层级和缩进。  
   - 代码块、行内代码（反引号包裹）、命令、文件名、路径、URL、占位符（如 `<your_name>`）均**不翻译**。  
   - 代码中的注释可以翻译，但必须保持代码缩进和注释符号不变。  
   - 数字、运算符、版本号、配置项键名等照原样保留。

3. **专有名词与引用**  
   - 产品名、公司名、商标、人名、项目名、库名、框架名一般保留原文，如果存在官方中文名则使用官方名（如 GitHub，不译；React，不译）。  
   - 引用的错误信息、日志输出，可选择性翻译，但必须在译文后用 `[原文: ...]` 注明原始内容，例如：`权限被拒绝 [原文: Permission denied]`。

4. **语言风格**  
   - 保持技术文档的正式、客观、简洁风格，杜绝口语化表达。  
   - 句子需通顺自然，符合中文技术文档的行文习惯（被动语态适当转化，长句合理切分）。  
   - 标题、步骤说明、警告、注意等保留原有语气和层级。

5. **特殊元素处理**  
   - 图片、图表的替代文本（alt text）需翻译其描述部分。  
   - 链接的显示文字需翻译，但链接地址不译，格式为 `[中文描述](原始URL)`。  
   - 原文中的强调标记（如粗体、斜体）需在译文中合理再现。

6. **不确定及错误处理**  
   - 若遇到明显拼写错误或上下文矛盾，请根据上下文推测原意并翻译，同时在译文后以 `[译注：原文疑为xxx]` 说明。  
   - 对实在无法确定的内容，保留原文并在其后用括号加问号标注，例如 `quux(?)`。

【输出要求】  
- 只输出翻译后的中文文档，不要添加任何额外的解释、总结或评价。  
- 维持原文的段落、空行和布局，方便直接替换使用。
