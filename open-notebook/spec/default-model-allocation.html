语言模型如何被选中

provision_langchain_model() 在调用 LangChain 前统一解析（仅 language 类型）。

flowchart TD Start([provision_langchain_model]) --> T{token_count 大于 105000} T -->|是| L[使用 large_context_model] T -->|否| I{存在 model_id} I -->|是| X[使用指定模型] I -->|否| D[get_default_model 按 default_type] D --> Types[chat / transformation / tools] Types --> FB{transformation 或 tools 未配置} FB -->|是| C[回退 default_chat_model] FB -->|否| OK[使用已配置槽位]
阈值与估算：105_000 写死在 open_notebook/ai/provision.py。token_count 使用 o200k_base 估算，与真实模型分词可能有偏差。

资料摄入管线

与 source_graph 一致：提取正文 → 保存 → 可选向量化 → 并行转换。

flowchart TD S([开始摄入]) --> STT{配置了 STT 默认} STT -->|是| A[写入 audio_provider 与 audio_model] STT -->|否| E A --> E[extract_content] E --> SV[save_source 持久化 full_text] SV --> EM{需要 embed} EM -->|是| V[vectorize 异步任务] EM -->|否| TR V --> TR[并行 run transformation 子图] TR --> Done([结束])

详见 spec/business-flows.md 第 2 节。

Ask（检索 + 综合）

多阶段 LangGraph：策略 JSON → 多路向量检索 → 分路摘要 → 最终答案。

flowchart TD Q[用户问题] --> N1[agent 节点] N1 --> N2[并行 provide_answer] N2 --> VS[vector_search 每路检索词] VS --> N3[各路 LLM 归纳] N3 --> N4[write_final_answer] N4 --> Out[最终回答]
前后端差异：后端三阶段在未传入 RunnableConfig 中的 strategy_model / answer_model / final_answer_model 时，会按 default_type = tools 解析默认模型（open_notebook/graphs/ask.py）。前端「询问与搜索」页默认把三阶段都设为 default_chat_model（frontend/src/app/(dashboard)/search/page.tsx），除非用户自定义。

推荐梯度（经济 → 均衡 → 高阶）

与仓库内 docs/5-CONFIGURATION/ai-providers.md 的表述对齐；以下为选型参考，非运行时强制。

聊天 gpt-4o-mini；gemini-1.5-flash；qwen-turbo；llama3.1:8b（Ollama） gpt-4o；gemini-2.0-flash-exp；claude-sonnet-4-5；qwen-plus；llama-3.3-70b-versatile（Groq） o1 / o1-mini；claude-opus-4-5；qwen-max；gemini-1.5-pro-latest 文档「OpenAI / Gemini / Anthropic / Qwen / Groq」推荐节

自建与本地

Ollama、LM Studio、OpenAI-Compatible：零或低 API 成本，但需自管延迟、显存/内存与模型质量。详见文档 Self-Hosted / Local 与 Custom OpenAI-Compatible 章节。

  • Ollama：文档推荐 llama3.1:8b 均衡、phi3:3.8b 极省资源、llama3.3:70b 高质量。
  • OpenRouter：文档示例「快/省」google/gemini-2.0-flash-exp，「质量」anthropic/claude-sonnet-4.5。
