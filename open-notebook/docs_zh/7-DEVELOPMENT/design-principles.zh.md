# 设计原则与项目愿景

本文档阐述指导 Open Notebook 开发的核心原则、愿景与设计哲学。所有贡献者在提出变更或新功能前应阅读并理解这些原则。

## 🎯 项目愿景

Open Notebook 旨在成为 **注重隐私、可自托管的 Google Notebook LM 替代方案**，使用户能够：

1. **拥有研究数据** — 完全控制数据存放位置与访问者
2. **选择 AI 提供商** — 自由使用任意 AI 提供商或在本地运行模型
3. **定制工作流** — 灵活适配不同研究需求
4. **随处访问工作** — 通过 Web UI、API 或集成

### Open Notebook 是什么

- 用于管理与理解内容的 **研究助手**
- 连接多种 AI 提供商的 **平台**
- 将数据置于您控制之下的 **隐私优先** 工具
- 具备 API 与定制选项的 **可扩展系统**

### Open Notebook 不是什么

- 文档编辑器（请使用 Google Docs、Notion 等）
- 文件存储系统（请使用 Dropbox、S3 等）
- 通用聊天机器人（请使用 ChatGPT、Claude 等）
- 整套工作流的替代品（它是工具箱中的一项工具）

## 🏗️ 核心设计原则

### 1. 隐私优先

**原则**：用户数据与研究默认应处于用户控制之下。

**实践**：
- 自托管部署是主要用例
- 未经明确选择加入，不进行遥测或分析
- 不硬依赖特定云服务
- 清晰说明数据流向

**决策示例**：
- ✅ 支持本地 Ollama 模型
- ✅ 可配置 AI 提供商选择
- ❌ 硬编码云服务集成
- ❌ 强制外部服务依赖

### 2. 简洁优于功能堆砌

**原则**：工具应易于理解与使用，即使意味着功能更少。

**实践**：
- 清晰、聚焦的 UI，分区明确
- 对多数用户有效的合理默认
- 高级功能置于可选配置之后
- 面向非技术用户的文档

**决策示例**：
- ✅ 三栏布局（来源、笔记、聊天）
- ✅ 开箱即用的默认模型
- ❌  upfront 用过多选项淹没用户
- ❌ 基础任务的复杂多步流程

### 3. API 优先架构

**原则**：所有功能应可通过 API 访问，不仅限于 UI。

**实践**：
- UI 调用与外部客户端相同的 API
- 具备 OpenAPI 文档的完整 REST API
- 无无法自动化的「仅 UI」功能
- 前后端清晰分离

**决策示例**：
- ✅ 带完整 API 文档的 FastAPI 后端
- ✅ 各端点一致的 API 模式
- ❌ UI 组件中的业务逻辑
- ❌ 需直接访问数据库的功能

### 4. 多提供商灵活性

**原则**：用户不应被锁定在单一 AI 提供商。

**实践**：
- 通过 Esperanto 库支持多种 AI 提供商
- 易于切换提供商与模型
- 清晰说明各提供商限制
- 提供商不可用时优雅降级

**决策示例**：
- ✅ 支持 18+ AI 提供商
- ✅ 按功能选择模型（聊天、嵌入、TTS）
- ❌ 仅适用于 OpenAI 的功能
- ❌ 针对特定提供商的硬编码 API 端点

### 5. 通过标准实现可扩展性

**原则**：系统应通过明确定义的接口扩展，而非 Fork。

**实践**：
- 转换与命令的插件系统
- 标准数据格式（JSON、Markdown）
- 架构中的清晰扩展点
- 常见定制场景的文档

**决策示例**：
- ✅ 自定义转换模板
- ✅ 后台命令系统
- ✅ Jinja2 提示模板
- ❌ 无扩展点的硬编码业务逻辑

### 6. 异步优先以提升性能

**原则**：长时间运行的操作不应阻塞用户界面或 API。

**实践**：
- 后端全程 async/await 模式
- 重负载的后台作业处理
- 状态更新与进度跟踪
- 优雅处理 AI 提供商慢响应

**决策示例**：
- ✅ 数据库操作使用 AsyncIO
- ✅ 播客生成的后台命令
- ✅ 聊天的流式响应
- ❌ API 端点中的同步阻塞操作

## 🎨 UI/UX 原则

### 聚焦内容，而非界面装饰

- 尽量减少 UI 杂乱与干扰
- 内容应占据大部分屏幕
- 控件在需要时出现，而非始终可见
- 各视图布局一致

### 渐进式披露

- 先展示简单选项，按需展示高级选项
- 勿用所有可能设置淹没新用户
- 提供适用于约 80% 用例的合理默认
- 高级功能可发现但不突兀

### 响应迅速

- 常见操作 UI 应感觉即时
- 耗时操作显示加载状态
- 尽可能缓存与优化
- 慢速连接下优雅降级

## 🔧 技术原则

### 清晰的关注点分离

**各层不应泄漏**：
- 前端不应了解数据库结构
- API 不应包含业务逻辑（委托给领域层）
- 领域模型不应了解 HTTP 请求
- 数据库层不应了解 AI 提供商

### 类型安全与校验

**尽早发现错误**：
- 所有 API 边界使用 Pydantic 模型
- Python 代码库全程类型注解
- 前端使用 TypeScript
- 在系统边界校验数据

### 测试重要部分

**聚焦有价值的测试**：
- 测试业务逻辑与领域模型
- 测试 API 契约与错误处理
- 不测试框架代码（FastAPI、React 等）
- 关键工作流的集成测试

### 数据库为唯一事实来源

**SurrealDB 是我们的单一事实来源**：
- 所有状态持久化于数据库
- 数据库层无业务逻辑
- 恰当使用 SurrealDB 特性（记录链接、查询）
- 所有模式变更使用模式迁移

## 🚫 应避免的反模式

### 功能蔓延

**表现**：
- 因「很酷」或「容易」而加功能
- 在常见用例完善前为边缘用例建功能
- 试图满足所有人

**为何避免**：
- 增加复杂度与维护负担
- 使工具更难学习与使用
- 稀释核心价值

**替代做法**：
- 聚焦核心用例
- 对不符合愿景的功能说不
- 为边缘用例构建扩展点

### 过早优化

**表现**：
- 在不知是否慢之前优化代码
- 未衡量影响就采用复杂缓存
- 为边际性能牺牲代码清晰度

**为何避免**：
- 代码更难理解与维护
- 优化错误目标
- 浪费开发时间

**替代做法**：
- 先测量，再优化
- 聚焦算法改进
- 性能变更前先剖析

### 过度工程

**表现**：
- 为「以后可能需要」构建抽象层
- 为 3 行函数实现设计模式
- 构建框架而非解决问题

**为何避免**：
- 增加贡献者认知负担
- 简单变更需改多处文件
- 掩盖实际业务逻辑

**替代做法**：
- 从简单开始，模式出现后再重构
- 优化可读性与清晰度
- 在简化而非复杂化时使用抽象

### 无迁移路径的破坏性变更

**表现**：
- 改数据库模式却无迁移脚本
- 修改 API 契约却无版本化
- 无弃用警告就移除功能

**为何避免**：
- 破坏现有安装
- 使用户与贡献者沮丧
- 造成维护噩梦

**替代做法**：
- 模式变更始终提供迁移脚本
- 移除前先弃用
- 清晰记录破坏性变更

## 🤝 决策框架

评估新功能或变更时，请自问：

### 1. 是否符合愿景？
- 是否帮助用户拥有研究数据？
- 是否支持隐私与自托管？
- 是否契合核心用例？

### 2. 是否遵循原则？
- 是否简单易用、易理解？
- 是否可通过 API 使用？
- 是否支持多提供商？
- 用户能否扩展？

### 3. 实现是否可靠？
- 是否保持关注点分离？
- 是否正确类型化与校验？
- 是否包含测试？
- 是否有文档？

### 4. 成本如何？
- 增加多少复杂度？
- 维护负担多大？
- 是否引入新依赖？
- 使用频率是否值得成本？

### 5. 是否有替代方案？
- 现有功能能否解决？
- 能否以插件或扩展实现？
- 是否应作为独立工具？

## 📚 原则驱动决策示例

### 为何从 Streamlit 迁移到 Next.js

**原则**：API 优先架构

**理由**：
- Streamlit 将 UI 与后端逻辑耦合
- 难以构建外部集成
- 对 API 行为控制有限
- Next.js + FastAPI 提供清晰分离

### 为何使用 Esperanto 对接 AI 提供商

**原则**：多提供商灵活性

**理由**：
- 抽象提供商特定细节
- 易于添加新提供商
- 跨提供商一致接口
- 无厂商锁定

### 为何有后台命令系统

**原则**：异步优先以提升性能

**理由**：
- 播客生成需数分钟
- 用户不应等待长操作
- 需要状态跟踪与错误处理
- 支持未来批处理

### 为何支持本地 Ollama

**原则**：隐私优先

**理由**：
- 支持完全离线运行
- 不向外部服务发送数据
- 硬件投入后对用户免费
- 符合自托管理念

## 🔄 原则的演进

原则并非一成不变。随项目成长与用户反馈，部分原则可能演进。但核心原则变更应：

1. **理由充分** — 清晰说明为何需要变更
2. **公开讨论** — 重大变更征求社区意见
3. **文档化** — 在本文件中更新并说明
4. **渐进** — 尽可能不以破坏性变更实施
